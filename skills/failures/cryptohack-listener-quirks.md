CryptoHack's socket challenge listener has two behaviours that bit me repeatedly:

### 1. `self.no_prompt = True` — server sends the FIRST challenge response on connect

Normally the listener sends `before_input` on connect and waits for our input before calling `Challenge.challenge()`. But when the Challenge class sets `self.no_prompt = True` (either in `__init__` or after returning a response), the listener **immediately calls `challenge()` with an empty/dummy message and flushes that response to the client alongside the greeting.**

Implication: on connect you receive TWO pieces in one burst — the greeting line and the first JSON response. If you naively `recv_line` once, you'll hand the first JSON line back to the client code as "the greeting" and then time out waiting for "the first response".

**How to apply:** If a challenge sets `no_prompt=True`, drain everything on connect (`recv_all` with a timeout instead of `recv_line`), split into lines, and take the first `{...}` line as the server's initial response. Your first `send_json` should be the PROVE-phase payload, not a placeholder.

This showed up in Honest Verifier Zero Knowledge and Too Honest (ZKP Sigma Protocol).

### 2. "Transition" messages appear on the round that TRIGGERED the transition, not the next one

Pattern: a Challenge holds a game state with a background counter (`num_spins`, `num_deals`, etc.) that gets set when some action EMPTIES the current resource. The response for that round is constructed AFTER the action, so it sees the just-set counter and includes a "new croupier" / "reshuffle the deck" message — **on the same round**.

Concretely (Nothing Up My Sleeve): `spin_wheel()` pops the last digit of the current rebase sequence. After the pop, `self.spins == []`, so it calls `change_croupier()` and sets `num_spins`. Then `challenge()` checks `if self.game.num_spins:` and prepends "Good evening, I will be your new croupier" to **this round's** message. That round's spin is the LAST spin of the OLD sequence — NOT the first of the new one.

I initially wrote boundary detection as "see 'new croupier' → start a new sublist and put this spin in it", which put the last digit of each sequence into the next sequence's bucket. Debugging symptom: seq 0 had 45 digits (appeared "unlucky") while seq 1's reconstructed value exceeded the 240-bit mask — mathematically impossible, confirming the split was wrong.

**Why:** The counter is set by side-effects of THIS round's action but is READ in the same round's response construction. There is no "round after" for the signal to appear.

**How to apply:** For boundary-signalling server messages, always trace whether the signal is generated by the action of THIS round or the state inherited from a prior round. Append the current round's spin to the CURRENT sequence, and if the msg contains the transition marker, start a new sublist for NEXT round's spin:

```python
sequences[-1].append(spin)        # first — this spin belongs to the old seq
if "new croupier" in msg:
    sequences.append([])          # then — next round starts a new seq
```

Validation trick: if you reconstruct an LSB-masked value (mask = 2^k - 1) and the result has MORE than k significant bits, the boundary logic is wrong — one sequence ate a digit from the neighbour.
